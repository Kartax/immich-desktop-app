# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A macOS **File Provider** app that surfaces a self-hosted **Immich** server in Finder
and in every file open/upload dialog. Files are listed as placeholders and downloaded
on demand. Read-only. Swift + SwiftUI. There is also an in-app **gallery window**
("View Gallery" in the tray menu) showing the All Photos timeline with lazy paging.
The Xcode project is generated by XcodeGen from `project.yml` (the `.xcodeproj` is
git-ignored and must be regenerated).

## Commands

```sh
# Regenerate the Xcode project after changing project.yml OR adding/removing source files
xcodegen generate

# Fast compile-check from the CLI without code signing (use this to validate edits)
xcodebuild -project ImmichDesktop.xcodeproj -scheme ImmichDesktop \
  -destination 'platform=macOS' build CODE_SIGNING_ALLOWED=NO

# Live extension logs (info-level is NOT persisted, so log show usually shows nothing —
# always use log stream to debug the extension)
log stream --level debug --predicate 'subsystem == "org.kartax.ImmichDesktop"'
```

- **Running** the app requires full Xcode + a signing Team (⌘R from Xcode). The
  CLI compile-check above does not need signing and is the normal inner loop here.
- Adding a new `.swift` file means re-running `xcodegen generate` (the generated
  pbxproj lists files explicitly).
- `project.yml` pins `DEVELOPMENT_TEAM` and defines a shared, runnable `ImmichDesktop`
  scheme, so signing and the Run target survive regeneration (no "Choose an app to
  run" dialog, no re-selecting the team in Xcode). In Xcode, run the **ImmichDesktop**
  scheme, not the extension.
- There are no tests.
- If `swiftc`/`xcodebuild` complains about the Xcode license during a parse/compile
  check, prepend `DEVELOPER_DIR=/Library/Developer/CommandLineTools`.

## Architecture

Two targets that share the `Shared/` sources:

- **`ImmichDesktop/`** — container app. Captures server URL + API key and manages the
  File Provider domain via `DomainManager`. Runs as a **menu bar app** (`LSUIElement`,
  no Dock icon), driven by **AppKit, not SwiftUI scenes**: the SwiftUI `App` body is
  an empty `Settings` scene; `AppDelegate` owns an `NSStatusItem` whose menu is
  rebuilt on every open (`NSMenuDelegate.menuNeedsUpdate`) — version row, update row,
  "View Gallery", "Open Settings…", "Quit". Windows are lazily created `NSWindow`s
  hosting SwiftUI via `NSHostingController` (`isReleasedWhenClosed = false`). The
  settings window is kept and reused across opens (`ContentView` self-closes via an
  injected `onClose` after Save & Activate); the **gallery window is deliberately
  discarded on close** (`windowWillClose` → nil) so its asset list + image cache are
  freed and reopening shows a fresh timeline. Domain lifecycle is tied to the app:
  Quit calls `DomainManager.deactivate()` (removes the domain → Immich disappears
  from Finder), launch calls `activate(reset: false)` if configured (re-adds, keeps
  cache), Save & Activate calls `activate(reset: true)` (full reset).
- **`FileProviderExt/`** — `NSFileProviderReplicatedExtension` +
  `NSFileProviderThumbnailing`. This is where Finder's reads turn into Immich calls.

Data flow: `ImmichClient` (REST, `x-api-key`) → `FileProviderExtension` callbacks →
Finder. The extension never sees the UI; the app never serves files to Finder, but it
does use `ImmichClient` directly (settings test, `ConnectionMonitor`, gallery).

### In-app gallery (`ImmichDesktop/Gallery/`)

"View Gallery" shows the All Photos timeline in-app. Its defining constraint (unlike
the File Provider path): **never materialize the whole library** —
`ImmichClient.assetsPage(page:size:)` fetches a SINGLE `/search/metadata` page
(newest first, no `withExif`), and `GalleryViewModel.loadMoreIfNeeded` loads the next
page only when a cell among the trailing ~40 loaded items appears. Duplicate asset
ids across pages (live uploads shift page boundaries mid-scroll) are dropped — they
would crash `ForEach`. `ThumbnailLoader` caches decoded `NSImage`s in a bounded
`NSCache`, one instance per window. The detail view uses
`thumbnail(id:size: .preview)` — never `fullsize`, older servers lack it — and
streams video via `AVURLAsset` + `AVURLAssetHTTPHeaderFieldsKey` (`x-api-key`),
falling back to `downloadOriginal` when streaming fails (e.g. key scope). The
"Jump to" menu (data: one `/timeline/buckets?size=MONTH` call) re-anchors paging
via `takenBefore` and resets the list — a jump costs one request, never the pages
in between; scrolling *up* past the anchor is deliberately unsupported (jump again
or pick "Latest"). A `generation` counter in the view model keeps stale in-flight
load-more tasks from appending to a reset list.

### Icons (three places, two kinds)

- **App icon** (`AppIcon` in `ImmichDesktop/Assets.xcassets`): the user's PNG; used in
  Dock/app contexts. The full-color raster.
- **Menu bar + Finder sidebar must match and can only be monochrome SF Symbols.** The
  tray uses an `NSStatusItem` with the SF Symbol `camera.aperture` as a template image
  (`AppDelegate.icon`); the Finder sidebar uses the *same* symbol via
  `CFBundleIcons → CFBundlePrimaryIcon → CFBundleSymbolName` in the **extension's**
  `Info.plist`. A raster image cannot be used for the sidebar. Keep the two symbol
  names in sync. (`MenuBarIcon.imageset` is legacy/unused.)

### Two things that are easy to get wrong

1. **App ↔ extension config sharing.** `AppConfig` stores `{serverURL, apiKey}` as a
   **JSON file in the App Group container** (`FileManager.containerURL(forSecurity...)`),
   NOT `UserDefaults(suiteName:)`. On macOS the sandboxed extension cannot reliably
   read App-Group `UserDefaults` (cfprefsd refuses it). Do not "simplify" this back to
   UserDefaults.

2. **Item identifier scheme** (`ItemID`). The virtual tree is encoded into
   `NSFileProviderItemIdentifier` strings with `|` separators:
   `timeline`, `year|2024`, `month|2024-03`, `album|<id>`,
   `asset|<parentRaw>|<assetId>`. The asset's parent is embedded so the SAME photo can
   exist under both an album and a month. `ItemID` parses assets first (from both ends)
   because `parentRaw` itself contains `|`.

### The folder tree

Root contains an **"All Photos"** timeline folder (Year → Month → assets, from
`/timeline/buckets?size=MONTH` + `POST /search/metadata`) plus one folder per
**album** (`/albums`, `/albums/{id}`). `ItemEnumerator` maps each container kind to
the right call; `FileProviderExtension.item(for:)` answers folder metadata without
network and asset/album metadata with a fetch. Originals are fetched lazily in
`fetchContents`; grid previews in `fetchThumbnails`.

## Non-obvious gotchas (all currently handled — keep them)

- **`withExif: true` in the metadata search** is required: without it Immich omits
  `exifInfo.fileSizeInByte`, the item reports size 0, and macOS then skips the content
  download entirely (0-byte files, Quick Look errors).
- **`itemVersion` is derived from size+filename**, not constant — otherwise a size
  change (e.g. 0 → real) is treated as cached and never updates.
- **`NSFileProviderThumbnailing` must be declared on the class**, not just implemented,
  or the system never calls `fetchThumbnails`.
- **ATS exception** (`NSAllowsArbitraryLoads` + `NSAllowsLocalNetworking`) in both
  `Info.plist`s, because the server is reached over http / a private CA.
- The active domain identifier is **`ImmichDesktop`** (stable — do not version-bump it
  as a recovery trick). `DomainManager.activate(reset: true)` tears every domain down
  with **`.removeAll`** and re-adds, which is the recovery for a stuck "signed out /
  abgemeldet" state. `activate(reset: false)` (launch) keeps our domain + cache but
  prunes any stale/foreign domains (e.g. a leftover `immich-v2` from older builds), so
  identifier changes migrate cleanly. If a stuck state survives even `.removeAll`, the
  root cause is `fileproviderd`, not the app — clear it with `killall fileproviderd`
  (dev only), not by renaming the identifier.

## Distribution (download page + releases, all in this repo)

This repo is public and is the single home for both the source and the distribution:

- **Download page** — a static `docs/index.html` (plain HTML, inline `<style>`, no
  build step) served via **GitHub Pages** at
  https://kartax.github.io/immich-desktop-app/. `.github/workflows/pages.yml` deploys
  `docs/` on every push to `main` that touches it (Pages Source must be set to
  "GitHub Actions" in repo settings). Assets in `docs/`: `finder.png` + `settings.png`
  + `gallery.png` (screenshots), `logo.png` + `logo@2x.png` (favicon/OG image),
  `robots.txt` + `sitemap.xml`. **`logo.png` / `logo@2x.png` are copies of the app icon**
  (`ImmichDesktop/Assets.xcassets/AppIcon.appiconset/icon_512.png` / `icon_1024.png`) —
  recopy them when the icon changes. The canonical URL is hard-coded across
  `index.html` (canonical/OG/Twitter/JSON-LD), `sitemap.xml`, and `robots.txt`; update
  all three if it ever changes.
- **Releases** — `scripts/release.sh vX.Y.Z` builds, signs, notarizes the `.dmg` and
  publishes it as a GitHub Release **in this same repo** (`gh release create`, which
  also creates the `vX.Y.Z` tag at the built commit). `UpdateChecker.swift` reads this
  repo's `releases/latest` for the in-app update check. (Older builds were distributed
  from a separate `immich-desktop-app-public` repo; that history is migrated here. Do
  not point new work back at the `-public` repo.)

## Conventions

- Bundle id prefix is `org.kartax`; App Group is `group.org.kartax.ImmichDesktop`.
- API key needs all four of `album.read`, `asset.read`, `asset.download`, `asset.view`,
  plus `person.read` for the Persons view. `asset.view` is required, not optional: it
  gates `GET /assets/{id}/thumbnail`, so without it the Finder grid previews fail
  (only metadata/browsing/download would work). The in-app gallery (thumbnails,
  previews, video playback) relies on the same scopes.
- Do not create git commits in this repo on the user's behalf.
